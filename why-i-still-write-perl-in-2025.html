Why I Still Write Perl in 2025

October 2025

Everyone asks why I still write Perl. The answer is simple: because awk isn't enough and Python is too much.

The Problem With Modern Development

Your IDE weighs 500MB. Mine is vim and spite.

Your project has 47,000 dependencies in node_modules. Mine has use strict and unresolved childhood issues.

You import six libraries to parse CSV. I write split /,/ and move on with my life.

What Perl Taught Me

Everything is a string. Until it isn't. Then it's still a string.

Context matters. @array in scalar context returns its length. In list context, it returns itself. This is either genius or madness. I've been writing Perl for years and still don't know which.

Regex is not optional. Other languages treat regex like a feature. Perl treats it like punctuation. You don't "use" regex in Perl. You are regex.

The func Tool

I wrote a CLI tool called func to manage throwaway scripts. It does one thing: creates timestamped Perl scripts with boilerplate and opens them in Sublime.

func scraper    # Creates scraper-20251005_143022.pl
func            # Runs the latest script

No config files. No YAML. No "project structure." Just scripts that solve problems and disappear.

This is what Perl is for. Not enterprise architecture. Not microservices. Just solving the problem in front of you and moving on.

Why This Matters

The industry optimizes for maintainability. Fair enough. But sometimes you need to parse 10GB of logs right now, and the "proper solution" involves Docker, Kubernetes, and three sprint planning meetings.

Perl doesn't care about your roadmap. It cares about getting the job done.

What I'm Building

  • CLI tools that don't need frameworks
  • Text processors that finish before Python loads its imports
  • Scripts that solve one problem and do it in 50 lines

No users. No metrics. No product-market fit. Just code that works.

The Vibe

I code for fun. The bugs are free. Production code pays rent. Fun code pays in dopamine and regex crimes.
